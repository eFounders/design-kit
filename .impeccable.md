# [PROJECT NAME] — Design Kit

## Design Context

### Users
Designers and developers using the design-system storybook/playground to explore tokens, components and rules, and to demo the system live (change brand color, dark mode, radius, density, font, icon library). The page consumes `design-system/tokens.css`, which follows the **shadcn/ui token contract** so production devs can plug real shadcn components straight in.

### Two layers — treat them differently
1. **Playground chrome** (control/parameter bar, left nav, page shell, section headers, rule callouts, canvas stage). This is *the tool itself*. Its identity is the **neutral, monochrome, utilitarian layout** — not a separate font. It follows the project's typography tokens so the Font selector is visible across the whole page.
2. **Showcased components** (buttons, fields, filters, table, badges, tabs, card/stat, AI blocks). These *document the design system* and must stay faithful to shadcn. Fix bugs and inconsistencies here; do not make them flashy or divergent. They must use ONLY `var(--…)` tokens from `tokens.css`.

### Brand Personality
Precise, calm, trustworthy. A craftsperson's instrument, not a marketing page. Three words: **technical, exact, quiet.**

### Aesthetic Direction (chrome only)
Technical / utilitarian, in the spirit of Linear and Vercel dashboards. Monochrome chrome (neutral surfaces, foreground-on-background toggles) so switching the demoed brand color never makes the toolbar flicker. Thin 1px rules, compact controls, mono micro-labels for meta information, scrollspy nav. The brand color appears only inside the showcased components and the color swatches — never as chrome decoration.

- **Chrome typography:** follows the project tokens — `--font-base` for UI text (driven by the Font selector) and `--font-mono` for the mono micro-labels (`--sb-font-ui` / `--sb-font-mono` are thin aliases of these). The chrome identity is the monochrome palette + layout, not a bespoke font.
- **Anti-references:** glassmorphism everywhere, neon-on-dark, purple gradients, gradient text, colored left-stripe callouts, everything-in-a-card.

### Design Principles
1. **Tool vs content.** The chrome has its own stable identity; the components show the DS. Never let one bleed into the other.
2. **No hard-coded values in components.** Components use only `tokens.css` semantic tokens; the chrome uses scoped `--sb-*` variables. `tokens.css` IS the per-project source of truth — it's edited *intentionally* to recolor/extend the DS (brand, font, radius, tags…). The rule is: never hardcode a value in a component instead of using (or adding) a token.
3. **Monochrome chrome, brand in content.** Changing the brand recolors the components, not the tool.
4. **1px, not stripes.** Structure comes from spacing and hairline borders, never colored side-stripes (a banned AI tell).
5. **Faithful components.** Showcased components mirror shadcn; fixes restore fidelity, they don't reinvent.
